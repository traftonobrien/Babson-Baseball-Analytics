# Phase 17.5: Supabase Migration — Context

**Gathered:** 2026-03-22
**Status:** Ready for planning
**Source:** Primer locked decisions + user confirmation

<domain>
## Phase Boundary

Migrate the database driver from neon-http to Supabase Postgres, enable RLS with team_id scoping, and replace the static `web/public/data/` publish workflow with Supabase Storage. Auth is explicitly out of scope — PT_PASSWORD and MECHANICS_PASSWORD stay as-is.

</domain>

<decisions>
## Implementation Decisions

### Database Driver
- Replace `@neondatabase/serverless` with `@supabase/supabase-js` (or `postgres` via Supabase connection string)
- Supabase project: `ojrkhiatugtrwavyeakk` (`ojrkhiatugtrwavyeakk.supabase.co`)
- Drizzle ORM stays — only the driver underneath it changes
- All existing Drizzle schema and query code is preserved unchanged

### Row Level Security
- Pattern: `SET LOCAL app.team_id = 'babson'` at session start + policies using `current_setting('app.team_id')`
- NOT Supabase Auth JWT claims — session variable pattern only
- Apply RLS policies to all five charting tables that received team_id in Phase 17-02
- No application-layer `WHERE team_id = ?` filtering required after RLS is in place

### Authentication
- **LOCKED OUT OF SCOPE**: No Supabase Auth migration in this phase
- PT_PASSWORD and MECHANICS_PASSWORD remain exactly as-is
- Supabase Auth is Phase 18 scope

### Storage
- Replace `web/public/data/` static file publish with Supabase Storage bucket
- Outing publish scripts write to a Supabase Storage bucket instead of copying files to `web/public/data/`
- Web app reads outing assets from Supabase Storage URLs instead of `/public/data/` paths
- Existing published data in `web/public/data/` may need migration or can be re-published

### Claude's Discretion
- Whether to use `@supabase/supabase-js` client vs. `postgres` package with Supabase connection string for Drizzle
- Migration strategy for existing Neon data (likely Supabase import or pg_dump/restore)
- Bucket naming convention and public vs. private bucket policy for outing assets
- Order of operations for cutover (staging branch vs. direct migration)

</decisions>
