# Phase 08.1: Charting UX and Mode Workflow - Context

**Gathered:** 2026-03-07
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 08.1 exists to redesign the live charting experience around the real scoring workflow staff wants to use. The boundary is not pilot packaging, backend sync hardening, or broader scoring scope. It is the live charting screen, the pitch-entry interaction model, and the split between `Live AB` practice charting and `Game` charting with an established lineup.

</domain>

<decisions>
## Implementation Decisions

### User Priority
- The app should replicate the physical charting workflow digitally rather than feeling like a generic data-entry form.
- The scorer must be able to digest information quickly and avoid accidental pitch submissions.
- Core charting workflow decisions need to be settled before returning to pilot/TestFlight hardening.

### Screen Layout
- The zone stays dominant and belongs in the top-left of the live charting screen.
- Matchup and game context move into a compact top utility bar rather than a large side card.
- The right side of the screen is the control stack: count and pitch totals at the top, pitch type beneath that, action beneath that, and a pending-pitch confirm area below.
- Pitch history should not live on the main screen; it should move into a hidden drawer.

### Pitch Entry Workflow
- Pitch entry should become `select -> review -> confirm`, not immediate save on action tap.
- Selecting an action should only build the pending pitch, not commit it.
- The screen should expose a clear confirm affordance and keep undo available for committed events.
- The main screen should still expose plate-appearance closeout when the engine says a PA is ready to close.

### Mode Model
- The product should have two charting modes: `Live AB` and `Game`.
- `Live AB` is for practice and controlled scenarios, and it should start each at-bat through a compact setup sheet where the scorer picks pitcher, hitter, inning, outs, and starting count.
- `Game` mode uses a real lineup and engine-driven game flow, but the scorer should still have inline override ability for corrections when reality and the app need to be reconciled.

### Pitch Totals and Pitch-Type Filtering
- The main charting screen should always show all three pitch totals: current pitcher total, current inning total, and full game total.
- When a Babson pitcher is selected, pitch type selection should show only that pitcher's known arsenal plus `Other`.
- Matchup context must always include the current pitcher and hitter.

### Claude's Discretion
- The exact visual treatment of the top utility bar, control cards, and confirm strip.
- Whether the hidden pitch-history drawer opens from a button, edge handle, or segmented control.
- The precise inline correction affordances for `Game` mode as long as they do not collapse the screen back into a free-form editing layout.

</decisions>

<specifics>
## Specific Ideas

- The live charting screen should read like a coach's workflow: identify matchup, place the pitch, choose the pitch type, choose the action, confirm the pitch, then close the PA only when needed.
- Pitch history is lower priority than inning context and pitch totals, so screen real estate should go to current information first.
- The workflow should be able to handle random hitters and manual counts in `Live AB` mode without forcing that flexibility into `Game` mode all the time.
- The pitcher arsenal should reduce clutter in the pitch-type picker instead of showing every possible pitch family for every pitcher.

</specifics>

<code_context>
## Existing Code Insights

### Current Layout Baseline
- `ios/PitchTracker/PitchTracker/Views/Charting/LiveChartingView.swift` currently uses a top zone panel with a bottom dock and a permanent matchup/history column, which no longer matches the desired top-left / top-right / confirm workflow.

### Workflow Gap
- `ios/PitchTracker/PitchTracker/Views/Charting/PitchResultControls.swift` currently records a pitch immediately when an action is tapped, which conflicts directly with the required confirm-before-save behavior.
- `ios/PitchTracker/PitchTracker/Models/ChartingState.swift` currently stores only pitch type, location, and bunt context, so it will need a richer pending-pitch state to support selection without immediate submission.

### Reusable/Relevant Assets
- `ZoneGridView.swift` already gives the right dominant spatial work surface and should stay the anchor of the screen.
- `GameStore.swift` already tracks inning, outs, count, active pitcher, lineup position, and active pitches, so pitch totals and matchup context can be derived without inventing a second engine.
- The app already knows the active pitcher's `playerId` and handedness, which gives a clean hook for arsenal-based pitch-type filtering once arsenal data is made available in the iPad flow.

### Known Gaps
- There is no current `Live AB` versus `Game` mode model in the app shell or store.
- `PitchHistoryList.swift` is currently a full-time card and should be demoted to a secondary surface.
- Pitcher-specific arsenal data is not yet wired into the iPad charting picker even though the broader project already has arsenal data infrastructure and canonical pitcher identity.

</code_context>
