# Phase 08.1 Plan 03 Summary

**Pitch entry is now confirm-first, totals stay visible on the main screen, and bootstrap pitchers carry arsenal-filtered pitch families into iOS**

## Accomplishments
- Extended `ChartingState` so pitch action selection now builds a pending pitch instead of immediately saving on tap.
- Added explicit confirm, clear, and undo controls to the charting surface and kept pitcher, inning, and game pitch totals visible at all times.
- Expanded the charting bootstrap contract so each pitcher carries arsenal-filtered pitch families, then used that data in iOS to reduce pitch-type picker clutter.
- Added regression coverage for seeded counts, pending confirmation behavior, `Live AB` session lifecycle, and the updated bootstrap payload.

## Decisions Made
- Reused the first pitch's stored `ballsBefore` and `strikesBefore` as the deterministic seed for count reconstruction so manual-start counts remain compatible with the existing reducer.
- Made the new SwiftData bootstrap arsenal field optional to avoid breaking existing on-device store migration.

## Next Phase Readiness
- The iPad charting flow is now much closer to real scorer use and is safe enough to carry into Phase 9 pilot hardening.
- Bootstrap pitch-family filtering is in place, so future pilot work can focus on sync fidelity, auth shell, and operator diagnostics rather than more picker clutter cleanup.
