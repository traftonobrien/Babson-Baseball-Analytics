# Phase 12.2: NCAA leaderboard sync redesign - Context

**Gathered:** 2026-03-16
**Status:** Ready for planning

<domain>
## Phase Boundary

This phase redesigns the NCAA leaderboard sync implementation behind the existing app contract. It replaces the browser-driven fetch pipeline with a direct HTTP parsing pipeline, adds better failure classification and metadata, and introduces a more realistic sync strategy for nightly vs full refreshes.

The phase does not redesign the leaderboard UI or change unrelated analytics/profile work. It is operational and data-pipeline work scoped to NCAA sync reliability, speed, and diagnosability.
</domain>

<decisions>
## Implementation Decisions

### Fetch Architecture
- Replace Chromote and Chrome-driven NCAA page loading with direct HTTP requests plus HTML parsing.
- Do not keep a browser runtime in the GitHub Actions workflow for this sync path.
- Separate season/team lookup, fetch, normalization, derivation, and publish responsibilities.

### Data Compatibility
- Preserve the existing app-facing JSON outputs and keep compatibility with the current consumers in `web/app/api/team-stats/route.ts` and `web/lib/college-stats/babsonPitchers.ts`.
- Reuse the current derived metrics where possible instead of rewriting trusted formulas.
- Keep Babson as a hard validation target when NCAA data exists.

### Lookup Source
- Prefer current `collegebaseball` lookup data for NCAA season/team identifiers because it covers 2026.

### Operational Policy
- Support two execution scopes: nightly incremental sync and manual/full rebuild.
- Nightly incremental sync should always include Babson plus teams seen in the latest published cache or prior failure metadata, with a scheduled weekly full rebuild as the discovery backstop.
- Distinguish empty early-season teams from true request/parser failures.
- A degraded but usable sync should warn rather than fail the full workflow if Babson and output publishing succeed.
- Degraded publish behavior is locked: publish healthy legs, keep the prior published JSON for degraded legs unchanged, record the degraded leg as stale in `meta.json`, and never overwrite a failed leg with mixed-generation partial output.

### Claude's Discretion
- Exact file/module decomposition inside the sync implementation.
- Exact retry/backoff and concurrency settings for direct HTTP requests.
- Exact fixture/test strategy for parser coverage.
</decisions>

<specifics>
## Specific Ideas

- Replace `fetch_html_with_browser()` and browser session setup in `scripts/sync_collegebaseball_leaderboard.R`.
- Remove Chrome installation from `.github/workflows/sync-college-stats-nightly.yml`.
- Add structured result categories such as `success`, `empty_stats`, `access_denied`, and `parse_failed`.
- Upgrade `meta.json` so maintainers can tell whether a bad night was caused by NCAA empties, parser drift, or access denial.
- Shift nightly sync away from full Division III sweeps when most teams still have no stats.
- Change workflow control flow so publish and health evaluation still run even when one sync leg is degraded.
</specifics>

<deferred>
## Deferred Ideas

- Paid or licensed external NCAA data vendors.
- Leaderboard UI changes.
- Broader player-profile or mechanics roadmap work.
</deferred>
