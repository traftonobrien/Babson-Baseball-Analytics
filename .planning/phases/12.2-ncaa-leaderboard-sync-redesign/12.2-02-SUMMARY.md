## What was built

### Incremental sync mode
`--sync-mode incremental` builds the team list from:
1. Teams present in the latest published batting/pitching JSON
2. Failed teams from prior meta.json (retry them)
3. Babson always-include
4. Optional `--watch-list` teams

Falls back to full sweep when no prior cache exists.

### Structured meta.json
Each stat-type leg now reports:
- `outcome_counts`: success / empty_stats / access_denied / parse_failed counts
- `babson_present`: boolean
- `failure_count` and `failures` list with team_id, team_name, status, detail
- `stale`: true when prior file was retained due to validation failure
- `sync_mode` at top level

### Degraded publish policy (in R script)
- If validation fails for a leg, prior file is kept unchanged, `stale: true` set in meta.json
- Hard exit (status 1) only when the R script encounters Babson-missing or zero-rows failure
- `any_hard_failure` tracked across legs; script exits 1 only if at least one leg truly failed

### GitHub Actions workflow rewrite (`.github/workflows/sync-college-stats-nightly.yml`)
- Chrome/Chromote: **removed entirely** (setup-chrome step gone, chromote not in install list)
- `httr` added explicitly to R install list
- `determine-mode` job: nightly → incremental, Sunday UTC → full, dispatch → user choice
- `sync-pitching` and `sync-batting` as separate parallel named jobs (not matrix)
- `publish` job runs `if: always()` after both sync jobs
- Downloads use `continue-on-error: true`; on failure the prior checked-out file is kept
- `PITCHING_OK`/`BATTING_OK` env vars drive Python meta.json with `freshly_synced`/`stale` flags
- GitHub Actions `::warning::` annotations for degraded legs
- Hard-fails publish only if both legs failed
