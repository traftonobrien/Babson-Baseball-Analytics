# Phase 12.2 PRD: NCAA Leaderboard Sync Redesign

**Date:** 2026-03-16
**Status:** Ready for planning
**Owner:** Codex

## Problem

The current NCAA leaderboard sync relies on Chromote and a headless Chrome session per team page. Recent runs show the workflow can succeed, but it is still operationally weak:

- End-to-end runtime is roughly one hour per scheduled run.
- The fetch path is browser-driven and fragile.
- Early-season empty-team states are mixed together with true request failures.
- The workflow attempts a full DIII sweep every night even when most teams still have no usable stats.

The app-side consumer contract is already working and should not be destabilized. The redesign should replace the fetch architecture without changing the downstream JSON contract consumed by the web app.

## Goal

Replace the NCAA sync pipeline with a direct HTTP-based fetch architecture that is materially faster, simpler, and easier to reason about than the current Chromote workflow, while preserving the output JSON shape used by the existing leaderboard and player-profile surfaces.

## Non-Goals

- Redesign the leaderboard UI.
- Change the public JSON schema consumed by the app unless strictly required.
- Introduce a paid external data vendor in this phase.
- Solve unrelated player-profile or mechanics backlog work.

## Users

- Internal coaching staff relying on the team stats leaderboard and player pages.
- Repo maintainers who need a stable nightly sync and actionable failure diagnostics.

## Requirements

### Functional

1. The sync must fetch NCAA batting and pitching team/player stats without using Chromote or a browser runtime.
2. The sync must continue to support the 2026 season and Division III.
3. The pipeline must preserve the app-facing output files:
   - `web/public/college-stats/batting-2026.json`
   - `web/public/college-stats/pitching-2026.json`
   - `web/public/college-stats/meta.json`
4. The output row shape must remain compatible with:
   - `web/app/api/team-stats/route.ts`
   - `web/lib/college-stats/babsonPitchers.ts`
   - existing leaderboard UI consumers
5. The sync must classify fetch outcomes distinctly:
   - success
   - empty_stats
   - access_denied
   - parse_failed
6. `meta.json` must capture enough structured diagnostics to tell the difference between early-season empties and true scraper regressions.
7. Babson must remain a hard validation target in both batting and pitching outputs when NCAA data exists for Babson.
8. The workflow must support two sync scopes:
   - nightly incremental sync
   - manual or scheduled full rebuild

### Performance / Operational

9. The redesign must remove Chrome setup and Chromote package/runtime requirements from GitHub Actions.
10. Nightly runtime should drop materially from the current roughly one-hour baseline.
11. Failures caused by empty early-season teams should not fail the whole workflow if Babson and output writes succeed.
12. The workflow should emit warnings for degraded freshness and fail only for real pipeline breakage.

### Maintainability

13. The new sync architecture must separate:
   - season/team lookup
   - fetch
   - normalization
   - derived metrics
   - publish / metadata
14. Fetch logic must be testable or at minimum isolatable enough to fixture-check parser behavior.
15. The implementation should reuse current derived-metric logic where possible instead of rewriting trusted formulas.

## Locked Decisions

- Use direct HTTP requests plus HTML parsing instead of a browser session.
- Keep `collegebaseball` lookup data for current NCAA season/team identifiers if still the most current local option for 2026.
- Preserve the current web JSON contract unless a compatibility adapter is introduced in the same phase.
- Introduce incremental nightly sync behavior rather than scraping all 381 Division III teams every night.
- Keep a full rebuild path available through workflow dispatch.
- Define nightly incremental sync concretely as: always include Babson; include every team present in the latest published batting/pitching cache; include teams recorded in the prior run's failure metadata; support an optional watch-list for manually pinned teams; and rely on a scheduled weekly full rebuild as the discovery backstop for newly active teams.
- Define degraded publish policy concretely as: never publish mixed-generation outputs by overwriting a failed leg with partial data; if one stat type is healthy and the other is degraded, publish the healthy artifact, retain the prior published artifact for the degraded leg unchanged, and mark that leg as stale/degraded in `meta.json`; fail only when both legs are unhealthy or Babson validation fails for the leg that should have data.

## Open Implementation Space

- Whether the fetch implementation stays in one script or is split into helper files.
- Exact retry/backoff policy and concurrency level for direct requests.
- Whether to add parser fixtures in R, Node, or both.
- Exact names for helper scripts used to validate parser fixtures and consumer compatibility.

## Proposed Scope Breakdown

1. Replace browser-driven fetch helpers with direct request + parse helpers.
2. Preserve and adapt row normalization/player-link extraction.
3. Keep current batting/pitching derived metrics and final JSON shape.
4. Add structured failure categorization and richer `meta.json`.
5. Add incremental vs full sync modes to the script/workflow.
6. Remove Chrome/Chromote from workflow setup.
7. Verify compatibility against existing app consumers.
8. Restructure workflow control flow so publish/metadata evaluation still runs on degraded sync nights.

## Acceptance Criteria

1. A scheduled nightly run completes without installing Chrome and writes valid batting/pitching JSON plus metadata.
2. A manual full rebuild path still exists and can process the complete Division III team universe.
3. App routes consuming team stats require no breaking changes, or compatibility changes are fully included in the same phase.
4. `meta.json` clearly reports counts for success, empty stats, access denied, and parse failures.
5. The workflow marks degraded but usable syncs as warnings rather than hard failures when Babson and publish steps succeed.
6. Repo maintainers can inspect logs and metadata to understand whether a bad run was caused by NCAA empties, parser drift, or access denial.

## Risks

- NCAA may still rate-limit or deny some direct requests.
- Team selection for nightly incremental sync may miss newly active teams if the heuristic is too narrow.
- HTML table structure on `stats.ncaa.org` may drift and require parser fixture maintenance.
- Existing validation assumptions may need to be tightened around Babson presence vs broad division coverage.
