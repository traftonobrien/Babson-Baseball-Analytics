## Summary

This phase should extend the existing `/players/[slug]` profile surface rather than creating a standalone charting player route. The right architecture is to treat Live AB as one more profile data domain alongside Trackman, Command, and Mechanics, but unlike those tabs it must support both pitcher and hitter perspectives from the same player identity.

The current profile implementation is still pitcher-first. `web/app/players/[slug]/page.tsx` loads only pitching leaderboard data for the season snapshot and hero state, and the page hero explicitly labels the surface `Pitching Profile`. `PlayerProfileTabs.tsx` already provides the tab shell and visual language needed for a Live AB tab, so the main implementation question is not navigation; it is how to build a role-aware charting profile model that can represent pitcher data, hitter data, or both without forking the profile route.

The shared analytics engine in `web/lib/charting/analytics.ts` already solves the hardest math problem. Phase 12.1 should focus on a profile-facing aggregation layer, identity mapping for charted hitter names, and a compact tab presentation that highlights recent sessions and meaningful per-player overview stats. The Live AB tab should not embed a leaderboard table. It should behave more like the Trackman and Command tabs: overview cards, counts, recent history, and route-out links into deeper charting surfaces where needed.

**Primary recommendation:** add a charting profile helper module that resolves one player slug into Live AB pitcher data and Live AB hitter data, then build a new `Live AB` tab in `PlayerProfileTabs.tsx` around that view model. Treat hitter support as a first-class planning concern because it is currently less mature than pitcher identity and profile sourcing.

## Current Codebase Findings

### Existing player profile architecture
- `web/app/players/[slug]/page.tsx` is the server route that loads the player registry entry, season snapshot, percentile data, Trackman sessions, Command outings, and Mechanics data.
- The page already contains both `PITCHING_KEYS` and `BATTING_KEYS`, but only the pitching data path is currently used to build `seasonStats` and `d3Percentiles`.
- The hero still renders `Pitching Profile`, even on a route that could theoretically become more general.

### Existing tab shell
- `web/app/players/[slug]/PlayerProfileTabs.tsx` is the natural integration point.
- Current tabs: `Overview`, `Trackman`, `Command`, `Mechanics`.
- Each tab follows the same pattern: a hub link or context card, a few stat blocks, and a recent-session/outing list.
- This pattern maps well to a Live AB profile tab.

### Existing charting analytics and surfaces
- `web/lib/charting/analytics.ts` already supports aggregated pitcher and hitter Live AB stats.
- `/charting/leaderboard` already proves the cross-session aggregation path and metric set for both roles.
- `/charting/games/[id]` already gives detailed per-session pitcher and hitter review, which can serve as the deeper destination linked from the profile tab.

### Identity and hitter risk
- Pitcher integration is comparatively straightforward because player profiles already resolve canonical pitcher identities and charting pitcher segments store `playerId`.
- Hitter integration is the harder part because the charting hitter aggregation currently keys off `hitterName`, while the player profile route is slug/id based.
- That means Phase 12.1 likely needs an explicit charting hitter-to-profile identity resolution step before the tab can be trustworthy for hitters.

## Architecture Recommendations

### 1. Add a charting profile view-model layer

Create a dedicated helper module that:
- resolves a profile slug to Live AB pitcher aggregates, hitter aggregates, or both,
- normalizes empty-state behavior,
- builds recent-session rows suitable for the profile tab,
- centralizes any hitter identity matching logic.

This keeps `web/app/players/[slug]/page.tsx` from accumulating charting-specific aggregation and fallback rules inline.

### 2. Keep profile integration inside the existing route

Do not create a separate charting player route as the primary deliverable. The new value of this phase is bringing Live AB into the canonical player development page.

Recommended route model:
- `/players/[slug]` remains the canonical profile route
- `?tab=live-ab` deep-links directly to the new tab
- deeper charting destinations can still use `/charting/games/[id]` or future drill-down routes

### 3. Use role-aware tab composition

The Live AB tab should be designed around role availability:
- pitcher-only players: pitcher overview only
- hitter-only players: hitter overview only
- two-way players: show both via a compact switcher or stacked sections

Because the current profile page is already dense, a lightweight internal segment/switcher inside the Live AB tab is likely cleaner than stacking two large sections by default.

### 4. Prefer profile summaries over table embeds

The tab should emphasize:
- sample size / sessions
- workload and core outcomes
- approach/contact indicators
- recent sessions list
- direct links to charting session detail or broader leaderboard context

This matches the existing profile-tab design language and avoids duplicating the global leaderboard UI.

### 5. Make hitter identity resolution explicit in the plan

Hitter support should not be treated as a minor extension of pitcher support. The plan should explicitly cover:
- how charted hitter names map onto profile slugs/registry entries,
- what fallback or empty-state behavior happens when identity matching is ambiguous,
- where those resolution rules live and how they are tested.

## Testing Strategy

The phase should be verifiable with the current web toolchain.

Recommended coverage:
- unit tests for the new charting profile helper / identity resolution module,
- route-adjacent tests for role-aware view-model shaping and empty states,
- focused render tests if the Live AB tab UI logic becomes conditional enough to warrant them,
- clean production build as the final gate.

Best quick command for the phase:
`npm --prefix web test -- --run lib/charting`

Likely full safety net:
`npm --prefix web test -- --run`

Build gate:
`npm --prefix web run build`

## Validation Architecture

Phase 12.1 can stay Nyquist-compliant if it treats the risky parts as first-class automated targets:
- hitter identity resolution,
- profile view-model shaping for pitcher-only / hitter-only / two-way / no-data states,
- stable links into existing charting destinations,
- clean tab rendering under the existing player-profile shell.

The highest-value validation is not browser automation. It is deterministic helper coverage plus a build that proves the new tab integrates cleanly with the current profile route.

## Recommended File Layout

```text
web/lib/charting/
  playerProfile.ts               # new profile-facing charting aggregation + identity helpers
  playerProfile.test.ts          # new tests for role-aware profile shaping

web/app/players/[slug]/
  page.tsx                       # extend loader to fetch charting profile data
  PlayerProfileTabs.tsx          # add Live AB tab + role-aware tab content

web/app/charting/
  # reuse existing routes for drill-out links where possible
```

If the tab UI becomes too large, route-local components under `web/app/players/[slug]/` would be a reasonable follow-on, but the data-shaping helper should exist either way.

## Risks and Mitigations

| Risk | Why it matters | Mitigation |
|------|----------------|------------|
| Hitter identity mismatch | Charting hitter stats are name-keyed while profiles are slug/id keyed | Make identity resolution a planned task with explicit tests and fallback behavior |
| Pitcher-first profile assumptions | Current page hero and season loader are still pitching-oriented | Plan a scoped profile-generalization step rather than hard-coding Live AB to pitchers |
| UI duplication with leaderboard | Copying tables into profiles would create a second heavy leaderboard surface | Use compact overview cards and recent-session lists instead |
| Formula drift | Profile pages could reimplement charting metrics incorrectly | Reuse `web/lib/charting/analytics.ts` through a dedicated profile helper |

---

## Recommendation

Plan the phase in three parts:
- a profile aggregation/identity wave,
- a Live AB tab and player-page integration wave,
- a validation/polish wave that locks in two-way and empty-state behavior.

That sequence keeps the hard data-model risk first and prevents UI work from papering over unresolved hitter identity problems.
