# External Integrations

**Analysis Date:** 2026-03-06

## APIs & External Services

**Babson / Sidearm boxscores:**
- Babson Athletics HTML boxscores are fetched and parsed by `scripts/import_boxscore.py` and `scripts/post_game_update.py`
  - Integration method: HTTP fetch + HTML parsing
  - Auth: none
  - Purpose: deterministic game-stat imports under `web/public/stats/`

**D3 Dashboard API:**
- Proxied through `web/app/api/d3db/[...path]/route.ts`
  - Auth: `D3_DASHBOARD_API_KEY`
  - Purpose: fetch Babson pitching leaderboard data and player identifiers
  - Consumers: `web/lib/d3db.ts`, team stats route, sync scripts

**MLB / Savant data:**
- `web/app/api/savant-clips/route.ts` fetches Savant CSV/video/play-by-play endpoints
  - Auth: none visible in repo
  - Purpose: external clip lookup and contextual pitch data

**TrackMan exports:**
- Imported from local PDF exports via `scripts/import_trackman_pdf.py`
  - Purpose: build static session summaries, pitch-type tables, and index files under `web/public/trackman/`
  - Dependency type: local operator workflow rather than live API calls

## Data Storage

**Primary dynamic database:**
- Neon Postgres via `@neondatabase/serverless`
  - Connection: `DATABASE_URL`
  - Client: `web/db/index.ts`
  - Schema: `web/db/schema.ts`
  - Migrations: `web/drizzle/` generated by Drizzle Kit

**Primary static storage:**
- Git-committed JSON, CSV, and media under `web/public/`
  - Command outings: `web/public/data/<playerId>/<dateId>/`
  - Stats: `web/public/stats/`
  - Trackman: `web/public/trackman/`
  - Mechanics: `web/public/mechanics/`

**Local working storage:**
- Raw and intermediate files in `outings/`, `output/`, `Mechanics Analysis/`, `frames/`, `input/`, and `models/`

## Authentication & Identity

**Portal auth:**
- Simple password-based gate via `web/app/api/login/route.ts`
  - Session storage: HTTP-only `pt_auth` cookie
  - Fallback secret currently hardcoded if env var absent

**Mechanics auth:**
- Separate password gate via `web/app/api/mechanics-login/route.ts`
  - Session storage: HTTP-only `pt_mechanics` cookie
  - Used to keep mechanics content more restricted than the rest of the portal

**Player identity:**
- Canonical baseball identity flows through `web/public/data/Arsenals.csv`, `web/data/players.json`, and Python helpers like `scripts/lib/canonical_players.py`

## Monitoring & Observability

**Analytics:**
- `@vercel/analytics` is included in the web package

**Logs:**
- API routes use `console.log` / `console.error`
- Python scripts print warnings and dry-run plans to stdout/stderr
- No dedicated error tracking service is visible in the repo

## CI/CD & Deployment

**Hosting:**
- The web app is structured for Vercel deployment
  - Static assets are committed to git
  - Dynamic APIs are implemented as Next.js route handlers

**CI / automation:**
- GitHub Actions workflow `.github/workflows/sync-d3-daily.yml`
  - Daily sync of D3 data
  - Commits updated static leaderboard files back to the repo

## Environment Configuration

**Development:**
- `web/.env.local` for database credentials
- Local git worktree stores generated static data directly
- Python virtualenv plus downloaded models/assets are assumed

**Production:**
- Production auth relies on env-backed passwords if configured
- Dynamic DB reads require working Neon credentials
- Static data freshness depends on manual scripts or GitHub Actions, not background workers in the app itself

## Webhooks & Callbacks

**Incoming:**
- None visible in the current repo

**Outgoing:**
- None visible as first-class webhook integrations
- Most external interactions are pull-based fetches or operator-run imports
