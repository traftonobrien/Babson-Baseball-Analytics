# Codebase Structure

**Analysis Date:** 2026-03-06

## Directory Layout

```text
pitch-tracker/
├── src/                    # Python domain logic for ingest, TrackMan, mechanics, and analytics
├── scripts/                # Operational entrypoints, migrations, sync jobs, and import runners
├── tests/                  # Python tests and fixtures
├── web/                    # Next.js portal, API routes, DB schema, and static public assets
├── docs/                   # Architecture docs, runbooks, and generated contracts
├── data/                   # Baseball reference CSV/YAML inputs
├── outings/                # Working outing artifacts before/alongside publication
├── output/                 # Generated exports and processing outputs
├── Mechanics Analysis/     # Source mechanics media and manual clip metadata
├── .planning/              # GSD planning artifacts
├── requirements.txt        # Python dependency manifest
├── pytest.ini              # Python test discovery config
└── package.json            # Root Tailwind/PostCSS workspace package
```

## Directory Purposes

**src/**
- Purpose: reusable Python logic
- Contains: CV pipelines, mechanics metrics, TrackMan parsing, ingest helpers
- Key files: `src/mechanics/`, `src/trackman_pdf/`, `src/ingest_manual/`
- Subdirectories: grouped by domain rather than by technical layer

**scripts/**
- Purpose: operator-facing commands and automation entrypoints
- Contains: imports, syncing, publishing, migration, validation, batch runners
- Key files: `scripts/post_game_update.py`, `scripts/import_trackman_pdf.py`, `scripts/sync_d3_leaderboard.py`
- Subdirectories: `scripts/lib/` for shared script helpers

**tests/**
- Purpose: Python regression and contract tests
- Contains: module tests plus fixture HTML/PDF/manual JSON
- Key files: `tests/test_post_game_update.py`, `tests/mechanics/`, `tests/fixtures/`
- Subdirectories: mirror major Python subsystems

**web/**
- Purpose: portal application and its deployment-facing assets
- Contains: `app/`, `lib/`, `db/`, `data/`, `public/`, `drizzle/`
- Key files: `web/package.json`, `web/app/layout.tsx`, `web/db/schema.ts`, `web/lib/dataIndex.ts`
- Subdirectories: `web/app/` for routes, `web/lib/` for shared UI logic, `web/public/` for committed content

**docs/**
- Purpose: active architectural and operational documentation
- Contains: runbooks, ownership rules, data contracts, generated docs
- Key files: `docs/architecture/overview.md`, `docs/runbooks/post_game_update.md`, `docs/generated/web_app_data_contract.md`

## Key File Locations

**Entry Points:**
- `scripts/*.py` - operator workflows and automation jobs
- `web/app/page.tsx` - portal home
- `web/app/api/**/route.ts` - HTTP endpoints for dynamic features and proxies

**Configuration:**
- `package.json` - root workspace tooling
- `web/package.json` - web scripts and dependencies
- `requirements.txt` - Python deps
- `web/next.config.ts`, `web/tsconfig.json`, `web/drizzle.config.ts`, `web/eslint.config.mjs`
- `pytest.ini` - Python tests

**Core Logic:**
- `src/` - reusable Python modules
- `web/lib/` - reusable TypeScript models, selectors, aggregators, and loaders
- `web/db/` - Neon/Drizzle schema and DB client

**Testing:**
- `tests/` - Python tests
- `web/lib/**/*.test.ts` - Vitest tests
- `tests/fixtures/` - fixture HTML/PDF inputs

**Documentation:**
- `docs/` - repo documentation
- `tasks/` - historical feature planning notes
- `.planning/` - GSD-generated project docs

## Naming Conventions

**Files:**
- Python modules use `snake_case.py`
- React route files follow Next.js conventions (`page.tsx`, `layout.tsx`, `loading.tsx`)
- Many React components use `PascalCase.tsx`
- TypeScript unit tests use `*.test.ts`

**Directories:**
- Domain/group-based directories are common (`src/mechanics`, `web/app/trackman`, `docs/runbooks`)
- `web/public/*` mirrors published product areas rather than source modules

**Special Patterns:**
- `web/app/**/[param]/page.tsx` for route segments
- `web/public/data/<playerId>/<dateId>/` for published command outings
- `web/public/stats/players-by-id/<playerId>/<season>/<gameId>.json` for imported game stats

## Where to Add New Code

**New pipeline/domain logic:**
- Primary code: `src/<domain>/`
- Script entrypoint: `scripts/`
- Tests: matching area under `tests/`

**New portal feature:**
- Route/UI: `web/app/`
- Shared logic/types: `web/lib/`
- Dynamic persistence: `web/db/` and `web/app/api/`
- Static contract docs if needed: `docs/`

**New published artifact contract:**
- Implementation: relevant script in `scripts/`
- Consumer logic: `web/lib/` and `web/app/`
- Contract reference: `docs/generated/` or `docs/web/`

## Special Directories

**web/public/**
- Purpose: committed runtime data and media served directly by the web app
- Source: generated by scripts and operator workflows
- Committed: Yes

**outings/**
- Purpose: working outing folders and intermediate results
- Source: local processing workflows
- Committed: Yes in this repo, though content is operational rather than pure source code

**output/**
- Purpose: generated exports and analysis outputs
- Source: scripts and pipeline runs
- Committed: mixed; treat as generated artifacts with care
